## [Unreleased]

### Added

- **AW-2349 Phase 8 — Example app: `LockerBloc` screen lock integration**
  Completes the AW-2349 screen lock detection feature by wiring `ScreenLockService` into `LockerBloc`. Registers `on<_ScreenLocked>(_onScreenLocked)` in the constructor (after `on<_BiometricKeyInvalidationDetected>`). Sets `_screenLockService.onScreenLockedCallback = _onScreenLockDetected` in the constructor alongside the existing `_timerService.onLockCallback` assignment. Implements `_onScreenLockDetected` (guards on `!isClosed && status == unlocked`, then adds `LockerEvent.screenLocked()`) and `_onScreenLocked` (guards on `status == unlocked`, then calls `_lockerRepository.lock()`). Key design decision: `_onScreenLocked` bypasses the `BiometricOperationState` guard that `_onLockRequested` enforces — a physical device lock is an explicit OS security action that takes priority over any in-progress biometric flow. Adds `_screenLockService.startListening()` in `_onInitialEntrySubmitted()` and `_refreshUnlockedState()` immediately after `_timerService.startTimer()`, and `_screenLockService.stopListening()` in `_onLockerStateChanged()` (case locked) immediately after `_timerService.stopTimer()`. The native listener is now active only while the locker is unlocked, incurring zero cost when locked. The constructor parameter, field, and `dispose()` call were already present from Phase 7 scope leakage; only the handler, callback, and lifecycle wiring are new. Sole file changed: `example/lib/features/locker/bloc/locker_bloc.dart`. No new files, no code generation needed.

- **AW-2349 Phase 7 — Example app: DI wiring + `screenLocked` event**
  Wired `ScreenLockService` through the example app's factory-based DI graph and declared the `LockerEvent.screenLocked()` Freezed event. `RepositoryFactoryImpl` gains two `late final` fields: `_biometricCipher` (instantiated as `BiometricCipher()` in `init()`) and `_screenLockService` (instantiated as `ScreenLockServiceImpl(biometricCipher: _biometricCipher)`); `dispose()` calls `_screenLockService.dispose()` synchronously before the async repository teardown. The `RepositoryFactory` abstract interface gains `ScreenLockService get screenLockService`. `BlocFactoryImpl` gains a `required ScreenLockService screenLockService` constructor parameter (stored as `_screenLockService`) and passes it to `LockerBloc(...)`. `main.dart` wires `repositoryFactory.screenLockService` into the `BlocFactoryImpl(...)` call. `LockerEvent.screenLocked()` is declared as a zero-argument `const factory` in `locker_event.dart` and code-generated into `locker_bloc.freezed.dart` via `make g`. `LockerBloc` receives the `screenLockService` parameter and disposes it in `close()` (partial Phase 8 scope — the event handler, callback, and `startListening()`/`stopListening()` wiring remain for Phase 8). No new tests added.

- **AW-2349 Phase 6 — Example app: `ScreenLockService`**
  Added `example/lib/core/services/screen_lock_service.dart` — the example app's service layer bridge between `BiometricCipher.screenLockStream` and the upcoming `LockerBloc` integration. `ScreenLockService` is a minimal abstract class with a callback setter (`onScreenLockedCallback`), `startListening()`, `stopListening()`, and `dispose()`. `ScreenLockServiceImpl` wraps `BiometricCipher.screenLockStream` in a managed `StreamSubscription<bool>?`; `startListening()` cancels any existing subscription before creating a new one (safe re-subscribe); `stopListening()` is idempotent; `dispose()` cancels the subscription and nulls the callback reference. The `bool` payload from the stream is discarded — only the event occurrence matters. The class mirrors `TimerService` in structure, requiring no new architectural concepts. No other files modified; DI wiring (Phase 7) and BLoC integration (Phase 8) are out of scope for this phase.

- **AW-2349 Phase 4 — Windows native: `ScreenLockStreamHandler` (C++)**
  Added the Windows C++ native handler that delivers events into the `EventChannel("biometric_cipher/screen_lock")` established in Phase 1. Two new files are created under `packages/biometric_cipher/windows/`: `handlers/screen_lock_stream_handler.cpp` and `include/biometric_cipher/handlers/screen_lock_stream_handler.h`. The class is placed in `namespace biometric_cipher`, consistent with all other headers in the plugin. Session lock detection uses `WTSRegisterSessionNotification(hwnd, NOTIFY_FOR_THIS_SESSION)` to subscribe to WTS session change events for the current user's session, and `RegisterTopLevelWindowProcDelegate` to hook into the Flutter window procedure. When `WM_WTSSESSION_CHANGE` arrives with `wParam == WTS_SESSION_LOCK`, `event_sink_->Success(flutter::EncodableValue(true))` is called. The delegate returns `std::nullopt` so other handlers continue to receive the message. WTS registration and window proc delegate registration are tied to the Dart stream subscription lifecycle (`onListen` / `onCancel`); the destructor calls `UnregisterWindowProc()` as a safety net for engine shutdown. A `window_proc_delegate_id_ = -1` sentinel prevents double-unregistration. `ScreenLockStreamHandler` is owned by a new public `screen_lock_handler_` field on `BiometricCipherPlugin`, constructed in the static `RegisterWithRegistrar` and assigned to the plugin instance before `AddPlugin`. `CMakeLists.txt` adds `handlers/screen_lock_stream_handler.cpp` to `PLUGIN_SOURCES` and links `Wtsapi32` via `target_link_libraries` for both the plugin target and the test runner target; no `#pragma comment` is used. No Dart files, no Android files, no Apple platform files, and no test files were modified. All changes scoped to `packages/biometric_cipher/windows/`.

- **AW-2349 Phase 3 — iOS/macOS native: `ScreenLockStreamHandler.swift`**
  Added the Apple-platform native handler that delivers events into the `EventChannel("biometric_cipher/screen_lock")` established in Phase 1. A new `ScreenLockStreamHandler.swift` file in `darwin/Classes/` implements `FlutterStreamHandler` with `NSObject` superclass using `#if os(iOS)` / `#elseif os(macOS)` compile-time guards. iOS observes `UIApplication.protectedDataWillBecomeUnavailableNotification` on `NotificationCenter.default` (public, App Store-safe); macOS observes `com.apple.screenIsLocked` on `DistributedNotificationCenter.default()`. Both notifications are delivered on the main thread by OS guarantee — no `DispatchQueue.main.async` wrapper needed. Observer registration is tied to `onListen` / `onCancel` using the target/selector (`#selector`) API. `BiometricCipherPlugin.register(with:)` registers `FlutterEventChannel("biometric_cipher/screen_lock")` using `registrar.messenger()` (iOS, method call) and `registrar.messenger` (macOS, property access), consistent with the existing `FlutterMethodChannel` pattern in the same method. `FlutterEventChannel.setStreamHandler(_:)` retains the handler — no new stored property on the plugin class. Import style uses `import Cocoa` on macOS (umbrella framework, functionally equivalent to `import AppKit`). No Dart files, no Android files, no Windows files, and no test files were modified. All changes scoped to `packages/biometric_cipher/darwin/Classes/`.

- **AW-2349 Phase 2 — Android native: `ScreenLockStreamHandler`**
  Added the Android native handler that delivers events into the `EventChannel("biometric_cipher/screen_lock")` established in Phase 1. A new `ScreenLockStreamHandler` class (in a new `handlers/` package directory) implements `EventChannel.StreamHandler` using a dynamically registered `BroadcastReceiver` for `ACTION_SCREEN_OFF`. The receiver is registered with the application context in `onListen` and unregistered in `onCancel`, tying the native listener lifetime to the Dart stream subscription. `BiometricCipherPlugin.onAttachedToEngine` instantiates and wires the `EventChannel` and stores both references as nullable class fields; `onDetachedFromEngine` calls `setStreamHandler(null)` and nullifies the fields. No Dart files, no iOS/macOS/Windows files, and no test files were modified. All changes scoped to `packages/biometric_cipher/android/`.

- **AW-2349 Phase 1 — Dart plugin: `screenLockStream` API surface**
  Added `screenLockStream` (`Stream<bool>`) to the `biometric_cipher` plugin across all three Dart layers. `BiometricCipherPlatform` gains a `screenLockStream` getter defaulting to `const Stream<bool>.empty()` for unsupported platforms. `MethodChannelBiometricCipher` adds a `static const EventChannel('biometric_cipher/screen_lock')` field and a `late final Stream<bool>` override backed by `receiveBroadcastStream().map((event) => event as bool)` — lazily initialized and cached so only one native subscription is opened regardless of listener count. `BiometricCipher` exposes the stream on the public facade with no `configure()` guard. `MockBiometricCipherPlatform` gains a public `StreamController<bool>.broadcast()` field, and `biometric_cipher_test.dart` gains a `group('screenLockStream', ...)` with three test cases. Pure Dart change; no native code touched. All changes scoped to `packages/biometric_cipher/`.

- **AW-2160 Phase 15 — Example app: proactive biometric key invalidation detection integration**
  Wired the example app to pass `biometricKeyTag: AppConstants.biometricKeyTag` in `LockerRepositoryImpl.determineBiometricState()`, activating the silent key validity probe introduced in Phase 13. `LockerBloc._determineBiometricStateAndEmit` (called at every lock-screen mount) stores `BiometricState.keyInvalidated` directly into `state.biometricState` when the library returns it. `LockedScreen` derives biometric button visibility from `state.canUseBiometric`, which returns `false` for `keyInvalidated`; `BiometricUnlockButton` adds an explicit `state.biometricState.isKeyInvalidated` guard as a defense-in-depth safety net. Both widgets rebuild via `buildWhen` on `biometricState` changes. The result: the lock screen renders with password-only UI from the first frame when the biometric key is invalidated — no button flash. The `BiometricState` enum is the single source of truth for invalidation state; no separate `isBiometricKeyInvalidated` boolean field was added. No library or plugin code changed. No new files created. Four existing files in `example/lib/features/locker/` were verified as correct; no code changes were required.

- **AW-2160 Phase 14 — Unit tests for proactive biometric key validity detection**
  Added 10 unit tests covering the three symbols introduced in Phase 13. `test/locker/models/biometric_state_test.dart` gains 5 tests: `keyInvalidated.isKeyInvalidated` returns `true`; `keyInvalidated.isEnabled` and `keyInvalidated.isAvailable` both return `false`; and regression guards confirming `enabled.isKeyInvalidated` and `availableButDisabled.isKeyInvalidated` return `false`. `test/security/biometric_cipher_provider_test.dart` gains 2 tests confirming `BiometricCipherProviderImpl.isKeyValid` delegates to `BiometricCipher.isKeyValid` and passes the return value through unchanged for both `true` and `false`. `test/locker/mfa_locker_test.dart` gains 3 tests: `determineBiometricState(biometricKeyTag:)` returns `keyInvalidated` when `isKeyValid` is `false` (with `verify` confirming delegation), returns `enabled` when `isKeyValid` is `true`, and never calls `isKeyValid` when no tag is supplied (confirmed by `verifyNever`). No production code changed. No new files created. All existing tests unbroken.

- **AW-2160 Phase 13 — Locker: `BiometricState.keyInvalidated` + proactive `determineBiometricState`**
  Added `BiometricState.keyInvalidated` enum value and `isKeyInvalidated` getter to the locker library. `keyInvalidated` is an error state (not included in `isAvailable` or `isEnabled`) returned when the hardware biometric key has been permanently invalidated after an enrollment change. Wired the silent key validity probe (built in Phases 9–12) into `MFALocker.determineBiometricState` via a new optional `biometricKeyTag` parameter: when provided and biometrics are enabled in settings, `isKeyValid(tag:)` is called without triggering a biometric prompt — if the key is gone, `keyInvalidated` is returned immediately. Backwards compatible: callers without `biometricKeyTag` retain existing behavior. Added `isKeyValid({required String tag})` to `BiometricCipherProvider` and `BiometricCipherProviderImpl`. Five files modified, no new files, no native changes.

- **AW-2160 Phase 12 — Dart plugin: `BiometricCipher.isKeyValid(tag)`**
  Wired the Dart-side bridge for the `isKeyValid` method channel call, completing the silent key validity probe stack built across Phases 9–11. `BiometricCipherPlatform` gains the abstract `isKeyValid({required String tag})` method; `BiometricCipher` gains the public API with an empty-tag guard (throws `BiometricCipherException(invalidArgument)` synchronously) and delegation to the platform interface. The method channel override calls `invokeMethod('isKeyValid', {'tag': tag})` with a `?? false` null guard and `PlatformException` mapping. Does not require the plugin to be configured — no biometric prompt is involved. Four automated tests added: `true` for an existing key, `false` for a nonexistent key, `false` after deletion, and the empty-tag guard. No native files touched, no new files created. Phase 13 (`BiometricCipherProvider.isKeyValid` + `determineBiometricState` integration) is now unblocked.

- **AW-2160 Phase 11 — Windows: `isKeyValid(tag)` silent probe**
  Added `IsKeyValidAsync(tag)` to the Windows C++/WinRT native layer of the `biometric_cipher` plugin. The method uses `KeyCredentialManager::OpenAsync(tag)` to probe whether a Windows Hello credential exists without triggering a biometric dialog. Returns `true` when the credential is accessible (`KeyCredentialStatus::Success`) and `false` for all other statuses including `NotFound`. Errors from WinRT exceptions surface as `PlatformException` to the Dart layer. The method name `"isKeyValid"` matches the Android (Phase 9) and iOS/macOS (Phase 10) handlers, completing native platform support for the Dart-side `invokeMethod` call planned in Phase 12. All changes are additions to existing files in `packages/biometric_cipher/windows/`; no Dart files, no new files, no logging added.

- **AW-2160 Phase 10 — iOS/macOS: `isKeyValid(tag)` silent probe**
  Added `isKeyValid(tag)` to the iOS/macOS Swift native layer. Uses `SecItemCopyMatching` with `kSecUseAuthenticationUISkip` to probe Secure Enclave key existence without triggering any authentication UI. Returns `false` when `errSecItemNotFound` — key has been deleted by the OS after a biometric enrollment change.

- **AW-2160 Phase 9 — Android: `isKeyValid(tag)` silent probe**
  Added `isKeyValid(tag)` to the Android Kotlin native layer. Uses `Cipher.init()` without a `BiometricPrompt` to probe key validity. `KeyPermanentlyInvalidatedException` is caught and mapped to `false`. Establishes the shared method name `"isKeyValid"` used across all platforms.

- **AW-2160 Phase 8 — Example app: password-only biometric disable recovery flow**
  Added `disableBiometricPasswordOnlyRequested` event, `_onDisableBiometricPasswordOnlyRequested` BLoC handler, and `disableBiometricPasswordOnly` repository method to the example app. When `isBiometricKeyInvalidated` is `true`, the Settings biometric toggle-off routes to the password-only path instead of a biometric prompt. The `isBiometricKeyInvalidated` flag is cleared on success and on successful biometric re-enable.

- **AW-2160 Phase 7 — Example app: biometric key invalidation UI**
  Added `isBiometricKeyInvalidated` flag to `LockerState`. Biometric unlock button and locked screen hide the biometric option when the flag is set. Auth bottom sheet shows an inline "Biometrics have changed" message. Settings screen shows an error-color description on the biometric tile. `biometricKeyInvalidated` side-effect action emitted from `_handleBiometricFailure`.

- **AW-2160 Phase 6 — Unit tests for Dart-layer key invalidation code paths**
  Added unit tests covering `BiometricCipherExceptionCode.keyPermanentlyInvalidated` mapping (Phase 3), `BiometricExceptionType.keyInvalidated` mapping (Phase 4), and `MFALocker.teardownBiometryPasswordOnly` (Phase 5).

- **AW-2160 Phase 5 — `MFALocker.teardownBiometryPasswordOnly`**
  Added `teardownBiometryPasswordOnly({required PasswordCipherFunc, required String biometricKeyTag})` to the `Locker` interface and `MFALocker` implementation. Removes the `Origin.bio` wrap using password authentication alone; hardware key deletion errors are suppressed because the key may already be gone.

- **AW-2160 Phase 4 — Locker library: `BiometricExceptionType.keyInvalidated`**
  Added `keyInvalidated` to `BiometricExceptionType`. `BiometricCipherProviderImpl` maps `BiometricCipherExceptionCode.keyPermanentlyInvalidated` to this new type.

- **AW-2160 Phase 3 — Dart plugin: `BiometricCipherExceptionCode.keyPermanentlyInvalidated`**
  Added `keyPermanentlyInvalidated` to `BiometricCipherExceptionCode` and mapped the platform error string `"KEY_PERMANENTLY_INVALIDATED"` to it in `fromString`.

- **AW-2160 Phase 2 — iOS/macOS native: `KEY_PERMANENTLY_INVALIDATED` error**
  iOS/macOS Swift layer emits `FlutterError(code: "KEY_PERMANENTLY_INVALIDATED")` when the Secure Enclave key is inaccessible after a biometric enrollment change, propagated through `KeychainServiceError`, `SecureEnclaveManagerError`, and `SecureEnclavePluginError`.

- **AW-2160 Phase 1 — Android native: `KEY_PERMANENTLY_INVALIDATED` error**
  Added `KEY_PERMANENTLY_INVALIDATED` to `ErrorType`. Android Kotlin layer catches `KeyPermanentlyInvalidatedException` in `executeOperation()` and emits `FlutterError(code: "KEY_PERMANENTLY_INVALIDATED")` to the Flutter channel.
