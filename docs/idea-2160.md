# Idea: Locker Package Changes for Biometric Change Handling (AW-2160)

## Summary

This document covers all changes required in the `mfa_locker` repository (a separate git dependency). It is a companion to `docs/idea-2160.md`, which covers the app-level wiring that consumes the APIs defined here.

The goal is to:
1. **Detect** when a biometric key has been permanently invalidated (as opposed to a generic auth failure), by exposing a dedicated `keyInvalidated` exception type through the `biometric_cipher` plugin and the `locker` layer.
2. **Clean up** the stale `Origin.bio` wrap when the hardware key is already gone, by providing a password-only teardown path in `MFALocker`.

---

## Affected Repository

`github.com/AdguardTeam/mfa_locker` — contains two sub-packages:
- `packages/biometric_cipher/` — Flutter plugin wrapping platform biometric + crypto APIs
- `lib/` — Dart locker logic (`MFALocker`, `LockerBiometricMixin`, security providers, exception types)

## A. `biometric_cipher` Native — Android

### Context: how exceptions flow

`SecureServiceImpl.decrypt()` calls `secureRepository.getCipher(secretKey, Cipher.DECRYPT_MODE, spec)`,
which calls `cipher.init()` **before** the `BiometricPrompt` is shown. When the Android KeyStore key has
been invalidated by a biometric enrollment change, `cipher.init()` throws
`KeyPermanentlyInvalidatedException` (a `java.security.GeneralSecurityException` subclass). This
exception propagates up through `SecureServiceImpl` into `SecureMethodCallHandlerImpl.executeOperation()`,
which already catches all `Exception`s. The catch block currently maps the error code as follows:

```kotlin
val errorCode = when (e) {
    is BaseException -> e.code          // typed plugin exception → uses its code field
    else             -> operationName   // anything else → falls back to method name (e.g. "decrypt")
}
```

Because `KeyPermanentlyInvalidatedException` is not a `BaseException`, it currently surfaces as error
code `"decrypt"` — indistinguishable from any other crash in the decrypt path.

Note: `BiometricPrompt.ERROR_KEY_PERMANENTLY_INVALIDATED` does **not** exist in the Android SDK (error
code 7 is `ERROR_LOCKOUT`). The exception is thrown from `Cipher.init()` _before_ the prompt is shown,
so `onAuthenticationError` is never involved.

**File**: `packages/biometric_cipher/android/src/main/kotlin/…/ErrorType.kt`

### Change 1: add `KEY_PERMANENTLY_INVALIDATED` to `ErrorType`


## B. `biometric_cipher` Native — iOS / macOS

### Context

On iOS and macOS, the Secure Enclave key is created with the `.biometryCurrentSet` access control flag. When any biometric change occurs (enrollment or removal), the hardware key becomes permanently inaccessible. Biometric invalidation manifests at **two distinct points**:

- **Point A** — `getPrivateKey` returns `nil`: `SecItemCopyMatching` with `LAContext` returns no result when the `.biometryCurrentSet` policy can no longer be satisfied. This can also happen on user cancel or lockout, so a secondary existence check is required to distinguish them.
- **Point B** — `SecKeyCreateDecryptedData` fails with `errSecAuthFailed (-25293)`: a key reference was obtained but the Secure Enclave refuses the cryptographic operation.

Note: `getPrivateKey` returns `SecKey?` — it does **not** throw. Wrapping it in a `do { try }` block is incorrect.

**File 2**: `packages/biometric_cipher/darwin/Classes/KeychainService.swift`

### Change 2: detect `errSecAuthFailed` in `decryptData()` and add `keyExists(tag:)` helper

```swift
func decryptData(_ data: Data, tag: String) throws -> Data {
    guard let key = getPrivateKey(tag: tag) else {
        // Key reference is nil — check whether the item still exists in the keychain.
        // If the OS deleted it (biometric change), keyExists returns false → permanently invalidated.
        // If it exists but auth failed (user cancel / lockout), surface a regular auth error.
        if !keyExists(tag: tag) {
            throw KeychainServiceError.keyPermanentlyInvalidated
        }
        throw KeychainServiceError.failedToDecryptData
    }

    var error: Unmanaged<CFError>?
    guard let decrypted = SecKeyCreateDecryptedData(
        key, .eciesEncryptionCofactorVariableIVX963SHA256AESGCM, data as CFData, &error
    ) as Data? else {
        if let cfError = error?.takeRetainedValue() {
            if CFErrorGetCode(cfError) == errSecAuthFailed {  // -25293
                throw KeychainServiceError.keyPermanentlyInvalidated
            }
        }
        throw KeychainServiceError.failedToDecryptData
    }
    return decrypted
}

/// Returns `true` if a Secure Enclave key item with the given tag exists in the keychain,
/// regardless of whether the caller can authenticate to use it.
private func keyExists(tag: String) -> Bool {
    guard let tagData = tag.data(using: .utf8) else { return false }
    let query: [CFString: Any] = [
        kSecClass: kSecClassKey,
        kSecAttrApplicationTag: tagData,
        kSecUseAuthenticationUI: kSecUseAuthenticationUISkip,
        kSecReturnAttributes: true,
    ]
    let status = SecItemCopyMatching(query as CFDictionary, nil)
    // errSecInteractionNotAllowed → item exists but requires auth → key is still there
    // errSecItemNotFound          → item gone (deleted by OS after biometric change)
    return status == errSecSuccess || status == errSecInteractionNotAllowed
}
```

### Change 4: propagate `KeychainServiceError.keyPermanentlyInvalidated` via re-throw

`getPrivateKey` returns `SecKey?` and does **not** throw. The secondary existence check lives inside `KeychainService.decryptData()` (Change 2). `SecureEnclaveManager.decrypt()` re-throws the new error type:

```swift
// Before:
func decrypt(data: Data, tag: String) throws -> Data {
    guard let key = getPrivateKey(tag: tag) else {
        throw SecureEnclaveManagerError.keyNotFound
    }
    return try keychainService.decryptData(data, tag: tag)
}

// After:
func decrypt(data: Data, tag: String) throws -> Data {
    do {
        return try keychainService.decryptData(data, tag: tag)
    } catch KeychainServiceError.keyPermanentlyInvalidated {
        throw SecureEnclaveManagerError.keyPermanentlyInvalidated
    }
    // Other KeychainServiceErrors propagate unchanged.
}
```

## E. `locker` Dart — `MFALocker` Password-Only Teardown

**File**: `lib/locker/mfa_locker.dart`

### Context

`MFALocker.teardownBiometry` currently requires both a `passwordCipherFunc` and a `bioCipherFunc`. The `bioCipherFunc` triggers a biometric prompt and uses the hardware key — which will fail if the key is already invalidated. The actual storage operation (`deleteWrap`) uses only `passwordCipherFunc`; the `bioCipherFunc` is only in the erasables list.

### Change: add `teardownBiometryPasswordOnly`

Add a new method that removes the `Origin.bio` wrap using password auth alone, then attempts to delete the hardware key with error suppression.

```dart
/// Removes the biometric wrap using password authentication only.
///
/// Use this when the biometric hardware key has been permanently invalidated
/// (e.g., after a biometric enrollment change) and [teardownBiometry] cannot
/// be called because the biometric prompt would fail.
///
/// Attempts to delete the hardware key identified by [biometricKeyTag] after
/// removing the wrap; errors during key deletion are suppressed because the
/// key may already be inaccessible or deleted by the OS.
Future<void> teardownBiometryPasswordOnly({
  required PasswordCipherFunc passwordCipherFunc,
  required String biometricKeyTag,
}) async {
  await _disableBiometryWithPasswordOnly(passwordCipherFunc: passwordCipherFunc);
  try {
    await _secureProvider.deleteKey(tag: biometricKeyTag);
  } catch (_) {
    // Key may already be gone or inaccessible — suppress.
  }
}

Future<void> _disableBiometryWithPasswordOnly({
  required PasswordCipherFunc passwordCipherFunc,
}) async {
  // Mirror the relevant parts of disableBiometry, omitting bioCipherFunc:
  // 1. Authenticate with password to get decrypted secret.
  // 2. Delete the Origin.bio wrap.
  // 3. Re-wrap under password only (no bio wrap written).
  // (Implementation follows existing disableBiometry logic, skipping bio cipher steps.)
}
```

**Alternative**: make `bioCipherFunc` optional in the existing `teardownBiometry` / `disableBiometry` methods:

```dart
Future<void> teardownBiometry({
  required PasswordCipherFunc passwordCipherFunc,
  BioCipherFunc? bioCipherFunc,  // nullable — skip bio erasable if null
  required String biometricKeyTag,
}) async { … }
```

The preferred approach (new method vs. optional parameter) is a locker-team decision. The app-level code in `docs/idea-2160.md` calls whichever API is chosen.

## F. Example App Integration

The example app (`example/lib/`) consumes the library-level APIs added in Sections C–E. The wiring follows the existing **UI → BLoC → Repository → MFALocker** architecture.

### Repository Layer

**File**: `example/lib/features/locker/data/repositories/locker_repository.dart`

Add a new method to both the `LockerRepository` interface and `LockerRepositoryImpl`:

```dart
/// Disable biometric authentication using password only.
///
/// Use when the biometric key has been permanently invalidated and the
/// normal [disableBiometric] flow (which requires a biometric prompt)
/// cannot succeed.
Future<void> disableBiometricPasswordOnly({required String password});
```

Implementation in `LockerRepositoryImpl`:

```dart
@override
Future<void> disableBiometricPasswordOnly({required String password}) async {
  await _ensureLockerInstance();
  final passwordCipherFunc = await _securityProvider.authenticatePassword(password: password);

  await _locker.teardownBiometryPasswordOnly(
    passwordCipherFunc: passwordCipherFunc,
    biometricKeyTag: AppConstants.biometricKeyTag,
  );
}
```

No `BioCipherFunc` is created — the entire operation is password-authenticated.

### BLoC State

**File**: `example/lib/features/locker/bloc/locker_state.dart`

Add a runtime flag to `LockerState`:

```dart
@freezed
abstract class LockerState with _$LockerState {
  const factory LockerState({
    // ... existing fields ...
    @Default(false) bool isBiometricKeyInvalidated,
  }) = _LockerState;
}
```

**Why a runtime flag?** `determineBiometricState()` still returns `BiometricState.enabled` because the hardware is fine and the `Origin.bio` wrap exists in storage. The invalidation is only discoverable at decrypt time when the platform throws `keyInvalidated`. The flag captures this runtime knowledge.

**Flag lifecycle:**
- **Set** when `_handleBiometricFailure` receives `BiometricExceptionType.keyInvalidated`
- **Cleared** on: successful `enableBiometric`, `eraseStorage`, or `disableBiometricPasswordOnly`

**File**: `example/lib/features/locker/bloc/locker_bloc.dart`

#### 1. Separate `keyInvalidated` from `failure` in `_handleBiometricFailure`

Currently `keyInvalidated` falls through to the generic `biometricAuthenticationFailed` action alongside `failure`. Change:

```dart
case BiometricExceptionType.failure:
case BiometricExceptionType.keyInvalidated:
  await _determineBiometricStateAndEmit(emit);
```

To:

```dart
case BiometricExceptionType.keyInvalidated:
  emit(state.copyWith(isBiometricKeyInvalidated: true));
  action(const LockerAction.biometricKeyInvalidated());
  // Reset to idle — no system dialog persists after key invalidation.
  add(
    const LockerEvent.biometricOperationStateChanged(
      biometricOperationState: BiometricOperationState.idle,
    ),
  );
  return;

case BiometricExceptionType.failure:
  await _determineBiometricStateAndEmit(emit);
```

The `keyInvalidated` case sets the flag, emits the dedicated action, resets to idle, and returns early — it does **not** fall through to the generic `biometricAuthenticationFailed` action.

#### 2. New handler: `_onDisableBiometricPasswordOnlyRequested`

```dart
on<_DisableBiometricPasswordOnlyRequested>(_onDisableBiometricPasswordOnlyRequested);
```

```dart
Future<void> _onDisableBiometricPasswordOnlyRequested(
  _DisableBiometricPasswordOnlyRequested event,
  Emitter<LockerState> emit,
) async {
  emit(state.copyWith(loadState: LoadState.loading));

  await _handleVaultOperation(
    operation: () async {
      await _lockerRepository.disableBiometricPasswordOnly(password: event.password);
      await _refreshBiometricState(emit, resetLoadState: true);

      if (!isClosed) {
        emit(state.copyWith(isBiometricKeyInvalidated: false));
      }

      action(const LockerAction.showSuccess(message: 'Biometric authentication disabled'));
    },
    onDecryptFailed: (error) => _handleDecryptFailure(
      emit,
      LockerAction.showError(message: 'Incorrect password: $error'),
    ),
    onError: (error) => _handleGenericFailure(
      emit,
      LockerAction.showError(message: 'Failed to disable biometric: $error'),
    ),
    operationDescription: 'disable biometric (password-only)',
  );
}
```

No `biometricOperationState` management — this is a password-only operation, no system biometric dialog is shown.

## Acceptance Criteria

1. `BiometricCipherExceptionCode.keyPermanentlyInvalidated` is thrown (not `decryptionError` / `authenticationError`) when the Android key is permanently invalidated.
2. `BiometricCipherExceptionCode.keyPermanentlyInvalidated` is thrown when the iOS/macOS Secure Enclave key is inaccessible due to a biometric enrollment change.
3. `BiometricExceptionType.keyInvalidated` is surfaced from `LockerBiometricMixin` (via `BiometricCipherProviderImpl`) when the underlying cipher code is `keyPermanentlyInvalidated`.
4. `MFALocker.teardownBiometryPasswordOnly` (or equivalent) successfully removes the `Origin.bio` wrap using password auth alone, without triggering a biometric prompt.
5. Generic auth failures (wrong fingerprint, lockout, user cancel) continue to produce `BiometricExceptionType.failure` / `cancelled` — they are not reclassified as `keyInvalidated`.
6. Example app detects `keyInvalidated` at runtime and displays "Biometrics have changed. Please use your password." inline in the auth bottom sheet.
7. Example app hides biometric unlock button on locked screen when key is invalidated.
8. Example app settings screen shows invalidation description and allows password-only biometric disable via `teardownBiometryPasswordOnly`.
9. Example app clears the invalidation flag on successful biometric enable, erase, or password-only disable.
