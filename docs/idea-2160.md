# Idea: Locker Package Changes for Biometric Change Handling (AW-2160)

## Summary

This document covers all changes required in the `mfa_locker` repository (a separate git dependency). It is a companion to `docs/idea-2160.md`, which covers the app-level wiring that consumes the APIs defined here.

The goal is to:
1. **Detect** when a biometric key has been permanently invalidated (as opposed to a generic auth failure), by exposing a dedicated `keyInvalidated` exception type through the `biometric_cipher` plugin and the `locker` layer.
2. **Clean up** the stale `Origin.bio` wrap when the hardware key is already gone, by providing a password-only teardown path in `MFALocker`.

---

## Affected Repository

`github.com/AdguardTeam/mfa_locker` — contains two sub-packages:
- `packages/biometric_cipher/` — Flutter plugin wrapping platform biometric + crypto APIs
- `lib/` — Dart locker logic (`MFALocker`, `LockerBiometricMixin`, security providers, exception types)

## A. `biometric_cipher` Native — Android

**File**: `packages/biometric_cipher/android/src/main/kotlin/…/SecureRepositoryImpl.kt`

### Change: catch `KeyPermanentlyInvalidatedException` in `getCipher()`

When the Android KeyStore key has been invalidated by a biometric enrollment change, `Cipher.init()` throws `KeyPermanentlyInvalidatedException`. Currently this propagates as a generic decryption error. It must be caught and mapped to a new, distinct error code.

```kotlin
// Before (simplified):
fun getCipher(keyAlias: String): Cipher {
    val cipher = Cipher.getInstance(TRANSFORMATION)
    val key = keyStore.getKey(keyAlias, null) as SecretKey
    cipher.init(Cipher.DECRYPT_MODE, key)  // throws KeyPermanentlyInvalidatedException here
    return cipher
}

// After:
import java.security.KeyPermanentlyInvalidatedException

fun getCipher(keyAlias: String): Cipher {
    val cipher = Cipher.getInstance(TRANSFORMATION)
    val key = keyStore.getKey(keyAlias, null) as SecretKey
    try {
        cipher.init(Cipher.DECRYPT_MODE, key)
    } catch (e: KeyPermanentlyInvalidatedException) {
        throw BiometricCipherException(ErrorType.KEY_PERMANENTLY_INVALIDATED, e.message)
    }
    return cipher
}
```

**File**: `packages/biometric_cipher/android/src/main/kotlin/…/AuthenticationRepositoryImpl.kt`

### Change: handle `ERROR_KEY_PERMANENTLY_INVALIDATED` in `onAuthenticationError`

`BiometricPrompt.ERROR_KEY_PERMANENTLY_INVALIDATED` (error code 7) is a distinct authentication error code that Android surfaces when the enrolled biometric set has changed. It currently falls into the generic `AUTHENTICATION_ERROR` branch.

```kotlin
// Before:
override fun onAuthenticationError(errorCode: Int, errString: CharSequence) {
    result.error(ErrorType.AUTHENTICATION_ERROR.code, errString.toString(), null)
}

// After:
override fun onAuthenticationError(errorCode: Int, errString: CharSequence) {
    val errorType = when (errorCode) {
        BiometricPrompt.ERROR_KEY_PERMANENTLY_INVALIDATED -> ErrorType.KEY_PERMANENTLY_INVALIDATED
        else -> ErrorType.AUTHENTICATION_ERROR
    }
    result.error(errorType.code, errString.toString(), null)
}
```

**File**: `packages/biometric_cipher/android/src/main/kotlin/…/ErrorType.kt` (or equivalent enum/sealed class)

### Change: add `KEY_PERMANENTLY_INVALIDATED` to `ErrorType`

```kotlin
enum class ErrorType(val code: String) {
    AUTHENTICATION_ERROR("AUTHENTICATION_ERROR"),
    DECRYPTION_ERROR("DECRYPTION_ERROR"),
    // ... existing entries ...
    KEY_PERMANENTLY_INVALIDATED("KEY_PERMANENTLY_INVALIDATED"),  // new
}
```

## B. `biometric_cipher` Native — iOS / macOS

**File**: `packages/biometric_cipher/darwin/Classes/KeychainService.swift` (or `SecureEnclaveManager.swift`)

### Context

On iOS and macOS, the Secure Enclave key is created with the `.biometryCurrentSet` access control flag, which causes the key to become inaccessible when any biometric is enrolled or removed. The failure surface is during decryption, not during key retrieval itself. The relevant OSStatus / LAError codes are:

- `errSecAuthFailed` (`-25293`) — generic auth failure; ambiguous
- `LAError.biometryNotEnrolled` — all biometrics removed (different from *changed*)
- `LAError.biometryLockout` — too many attempts
- The `.biometryCurrentSet` violation manifests as `errSecAuthFailed` with underlying `LAError` `.appCancel` or a CoreFoundation error — the reliable check is to attempt decryption and detect that it fails in a way that key deletion + re-create would fix

**Recommended approach**: after catching a decryption failure, attempt to retrieve the private key. If it is present but decryption fails, the key is invalidated. If the key is absent (deleted by the OS after biometric change), treat that as invalidated too.

**File**: `packages/biometric_cipher/darwin/Classes/SecureEnclaveManager.swift`

### Change: distinguish key-not-found-or-invalidated from generic failure

```swift
// Before:
func decrypt(data: Data, tag: String) throws -> Data {
    guard let key = try getPrivateKey(tag: tag) else {
        throw BiometricCipherPluginError.failedToDecryptData
    }
    // ... decrypt ...
}

// After:
func decrypt(data: Data, tag: String) throws -> Data {
    let key: SecKey?
    do {
        key = try getPrivateKey(tag: tag)
    } catch {
        // Key retrieval failed — treat as permanently invalidated if no key exists
        throw BiometricCipherPluginError.keyPermanentlyInvalidated
    }
    guard let privateKey = key else {
        // Key was deleted by the OS (biometric enrollment changed)
        throw BiometricCipherPluginError.keyPermanentlyInvalidated
    }
    do {
        return try performDecryption(key: privateKey, data: data)
    } catch let decryptError as CFError {
        // Map known OSStatus errors for biometry-set change to keyPermanentlyInvalidated.
        // errSecAuthFailed (-25293) after biometryCurrentSet violation is the typical signal.
        if isBiometryInvalidationError(decryptError) {
            throw BiometricCipherPluginError.keyPermanentlyInvalidated
        }
        throw BiometricCipherPluginError.failedToDecryptData
    }
}

private func isBiometryInvalidationError(_ error: CFError) -> Bool {
    // errSecAuthFailed = -25293
    let errSecAuthFailed: OSStatus = -25293
    return CFErrorGetCode(error) == errSecAuthFailed
}
```

**File**: `packages/biometric_cipher/darwin/Classes/BiometricCipherPlugin.swift`

### Change: add `keyPermanentlyInvalidated` case and map to channel error code

```swift
// In BiometricCipherPluginError (or equivalent):
enum BiometricCipherPluginError: Error {
    case failedToDecryptData
    // ... existing cases ...
    case keyPermanentlyInvalidated  // new
}

// In the plugin method channel handler, map the new case:
catch BiometricCipherPluginError.keyPermanentlyInvalidated {
    result(FlutterError(code: "KEY_PERMANENTLY_INVALIDATED",
                        message: "Biometric key has been permanently invalidated",
                        details: nil))
}
```

## E. `locker` Dart — `MFALocker` Password-Only Teardown

**File**: `lib/locker/mfa_locker.dart`

### Context

`MFALocker.teardownBiometry` currently requires both a `passwordCipherFunc` and a `bioCipherFunc`. The `bioCipherFunc` triggers a biometric prompt and uses the hardware key — which will fail if the key is already invalidated. The actual storage operation (`deleteWrap`) uses only `passwordCipherFunc`; the `bioCipherFunc` is only in the erasables list.

### Change: add `teardownBiometryPasswordOnly`

Add a new method that removes the `Origin.bio` wrap using password auth alone, then attempts to delete the hardware key with error suppression.

```dart
/// Removes the biometric wrap using password authentication only.
///
/// Use this when the biometric hardware key has been permanently invalidated
/// (e.g., after a biometric enrollment change) and [teardownBiometry] cannot
/// be called because the biometric prompt would fail.
///
/// Attempts to delete the hardware key identified by [biometricKeyTag] after
/// removing the wrap; errors during key deletion are suppressed because the
/// key may already be inaccessible or deleted by the OS.
Future<void> teardownBiometryPasswordOnly({
  required PasswordCipherFunc passwordCipherFunc,
  required String biometricKeyTag,
}) async {
  await _disableBiometryWithPasswordOnly(passwordCipherFunc: passwordCipherFunc);
  try {
    await _secureProvider.deleteKey(tag: biometricKeyTag);
  } catch (_) {
    // Key may already be gone or inaccessible — suppress.
  }
}

Future<void> _disableBiometryWithPasswordOnly({
  required PasswordCipherFunc passwordCipherFunc,
}) async {
  // Mirror the relevant parts of disableBiometry, omitting bioCipherFunc:
  // 1. Authenticate with password to get decrypted secret.
  // 2. Delete the Origin.bio wrap.
  // 3. Re-wrap under password only (no bio wrap written).
  // (Implementation follows existing disableBiometry logic, skipping bio cipher steps.)
}
```

**Alternative**: make `bioCipherFunc` optional in the existing `teardownBiometry` / `disableBiometry` methods:

```dart
Future<void> teardownBiometry({
  required PasswordCipherFunc passwordCipherFunc,
  BioCipherFunc? bioCipherFunc,  // nullable — skip bio erasable if null
  required String biometricKeyTag,
}) async { … }
```

The preferred approach (new method vs. optional parameter) is a locker-team decision. The app-level code in `docs/idea-2160.md` calls whichever API is chosen.

## Acceptance Criteria

1. `BiometricCipherExceptionCode.keyPermanentlyInvalidated` is thrown (not `decryptionError` / `authenticationError`) when the Android key is permanently invalidated.
2. `BiometricCipherExceptionCode.keyPermanentlyInvalidated` is thrown when the iOS/macOS Secure Enclave key is inaccessible due to a biometric enrollment change.
3. `BiometricExceptionType.keyInvalidated` is surfaced from `LockerBiometricMixin` (via `BiometricCipherProviderImpl`) when the underlying cipher code is `keyPermanentlyInvalidated`.
4. `MFALocker.teardownBiometryPasswordOnly` (or equivalent) successfully removes the `Origin.bio` wrap using password auth alone, without triggering a biometric prompt.
5. Generic auth failures (wrong fingerprint, lockout, user cancel) continue to produce `BiometricExceptionType.failure` / `cancelled` — they are not reclassified as `keyInvalidated`.
