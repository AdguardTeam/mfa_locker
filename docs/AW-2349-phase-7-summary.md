## Why This Was Needed

Phases 1–6 built the full plugin layer (native screen lock detection on Android, iOS, macOS, Windows) and created `ScreenLockService` as a Dart service class. However, the service had no consumers: it was not instantiated, not wired into any DI graph, and no BLoC event existed to represent a screen lock signal.

Phase 7 creates those connections:

- `RepositoryFactoryImpl` is the right place to own the `ScreenLockServiceImpl` instance because it already owns other long-lived services (`TimerService`) and is responsible for lifecycle management (creation in `init()`, teardown in `dispose()`).
- `BlocFactoryImpl` must receive `ScreenLockService` to pass it to `LockerBloc` — following the same injection path that `TimerService` already uses.
- `LockerEvent.screenLocked()` must be declared and code-generated before Phase 8 can register a handler for it.

Without Phase 7, `LockerBloc` has no way to receive screen lock signals, and the event type needed to express a screen lock trigger does not exist.

## Files Changed

### `example/lib/di/factories/repository_factory.dart`

Two new `late final` fields were added to `RepositoryFactoryImpl`:

- `late final BiometricCipher _biometricCipher` — a dedicated `BiometricCipher` instance for the screen lock stream. (The class had no `BiometricCipher` field before; the PRD confirmed this and required one to be introduced here.)
- `late final ScreenLockService _screenLockService` — instantiated as `ScreenLockServiceImpl(biometricCipher: _biometricCipher)`.

Both are initialized in `init()` in this order: `_timerService` (existing) → `_biometricCipher` (new) → `_screenLockService` (new) → `configureBiometricCipher(...)` (existing).

The `RepositoryFactory` abstract interface gains `ScreenLockService get screenLockService;` so that callers (`main.dart`) can access the service without depending on the concrete impl type.

`dispose()` now calls `_screenLockService.dispose()` synchronously before `await _lockerRepository?.dispose()`. This cancels the native stream subscription on app teardown and prevents a resource leak.

Note: the existing `_timerService` uses a nullable type with a `StateError` guard, while the new `_screenLockService` and `_biometricCipher` use `late final`. This is an intentional inconsistency acknowledged in the PRD — `late final` provides a clear crash (`LateInitializationError`) on misuse, which is equivalent in effect. `init()` is always called in `main.dart` before any getter is accessed.

### `example/lib/di/factories/bloc_factory.dart`

`BlocFactoryImpl` gains `required ScreenLockService screenLockService` as a constructor parameter (placed after `timerService`), stored as `final ScreenLockService _screenLockService`. The field is passed to `LockerBloc(...)` in the `lockerBloc` getter (see Deviations — this forwarding was planned for Phase 8, but was included here).

The `const` keyword was not removed from the constructor. The PRD and plan both explicitly required removing it on the grounds that `ScreenLockService` is an abstract interface. In practice, Dart accepts `const` on constructors whose fields are abstract-typed, as long as no call site uses `const`. No call site does, so the app compiles. The deviation is currently harmless but technically violates the stated constraint (see NC-6 in the QA report).

### `example/lib/main.dart`

`BlocFactoryImpl(...)` at line 39–43 now receives `screenLockService: repositoryFactory.screenLockService` (line 41). Parameter order at the call site is `lockerRepository`, `screenLockService`, `timerService` — valid because Dart named parameters are order-independent.

### `example/lib/features/locker/bloc/locker_event.dart`

Added as the final event in the sealed Freezed class:

```dart
/// Device screen was locked. Triggers immediate locker lock.
const factory LockerEvent.screenLocked() = _ScreenLocked;
```

Style matches all existing zero-argument events (`lockRequested`, `appResumed`, `activityDetected`).

### `example/lib/features/locker/bloc/locker_bloc.freezed.dart` (generated)

Regenerated by `cd example && make g`. The `_ScreenLocked` class is present with correct `==`, `hashCode`, `toString`, `map`, `when`, and their nullable variants. The event is immediately usable in Phase 8 handler registration.

## Deviations from Plan

### Deviation 1 — `const` not removed from `BlocFactoryImpl` (low risk)

The PRD and plan required removing `const` from the `BlocFactoryImpl` constructor as a forward-compatibility measure. The `const` keyword remains at line 20 of `bloc_factory.dart`. No existing or future call site uses `const BlocFactoryImpl(...)`, so this is harmless today. It should be corrected before Phase 8 to prevent surprises if a `const` call site is added by future contributors.

### Deviation 2 — `ScreenLockService` forwarded to `LockerBloc(...)` early (Phase 8 scope)

The plan explicitly deferred passing `_screenLockService` to `LockerBloc(...)` inside `BlocFactoryImpl.lockerBloc` to Phase 8 (because `LockerBloc` did not yet accept the parameter). In practice, the `LockerBloc` constructor was updated in the same commit, so `BlocFactoryImpl.lockerBloc` already passes `screenLockService: _screenLockService`. The app compiles and runs correctly.

### Deviation 3 — `locker_bloc.dart` modified (partial Phase 8 scope leakage)

`locker_bloc.dart` — which the PRD lists as a Phase 8 file — was modified during Phase 7:

- `import 'package:mfa_demo/core/services/screen_lock_service.dart'` added (line 11).
- `final ScreenLockService _screenLockService` field added (line 24).
- `required ScreenLockService screenLockService` constructor parameter added (line 29).
- `_screenLockService = screenLockService` in the initializer list (line 32).
- `_screenLockService.dispose()` called in `close()` (line 71).

What is NOT yet present (still Phase 8 scope):
- `on<_ScreenLocked>` event handler registration.
- `_onScreenLocked` event handler method.
- `_onScreenLockDetected` callback method.
- `_screenLockService.onScreenLockedCallback = ...` assignment.
- `_screenLockService.startListening()` / `stopListening()` calls on lock state transitions.

The partial implementation is architecturally safe: `LockerBloc` stores and disposes the service, but `startListening()` is never called, so the native subscription never opens, and no `on<_ScreenLocked>` handler is registered, so any screen lock event dispatched to the BLoC would be silently dropped by `flutter_bloc`. Existing behavior is completely unchanged. However, Phase 8 must be scoped carefully to add only the missing handler parts — the constructor injection is already done and must not be duplicated.

### Double-dispose safety

Because `LockerBloc.close()` calls `_screenLockService.dispose()` and `RepositoryFactoryImpl.dispose()` also calls `_screenLockService.dispose()`, the same instance is disposed twice on app teardown. `ScreenLockServiceImpl.dispose()` calls `stopListening()` which does `_subscription?.cancel(); _subscription = null` — both operations are null-safe and idempotent. Double-dispose is safe.

## DI Chain (Phase 7 contribution)

```
main()
  → RepositoryFactoryImpl(storageFilePath)
  → repositoryFactory.init()
       → _timerService = TimerServiceImpl(...)           [existing]
       → _biometricCipher = BiometricCipher()            [new]
       → _screenLockService = ScreenLockServiceImpl(     [new]
            biometricCipher: _biometricCipher)
       → configureBiometricCipher(...)                   [existing]
  → BlocFactoryImpl(
       lockerRepository: ...,
       timerService: ...,
       screenLockService: repositoryFactory.screenLockService,  [new]
     )
  → BlocFactoryImpl stores _screenLockService, passes to LockerBloc
  → LockerBloc stores _screenLockService, disposes on close()
```

```
repositoryFactory.dispose()
  → _timerService?.dispose()          [existing]
  → _screenLockService.dispose()      [new — synchronous]
  → await _lockerRepository?.dispose() [existing]
```

## QA Verdict

**RELEASE WITH RESERVATIONS**

The four core tasks (repository factory wiring, `main.dart` update, `LockerEvent.screenLocked()` declaration, code generation) are correctly implemented. The DI chain is complete and correct. Disposal is wired without leaks. The Freezed event is generated and ready.

Reservations that must be addressed before or during Phase 8:

1. **Static analysis (MC-1) was not run during the review session.** This is the designated acceptance criterion. `cd example && fvm flutter analyze --fatal-warnings --fatal-infos --no-pub .` must exit with code 0 before merge.
2. **`const` not removed from `BlocFactoryImpl`** — low risk today, should be corrected.
3. **`locker_bloc.dart` modified outside scope** — Phase 8 developer must be briefed: constructor injection is already done; only the event handler, callback, and start/stop wiring remain to be added.
4. **`LockerBloc` test files may need updating** — the new required `screenLockService` parameter on `LockerBloc(...)` may break any test that constructs `LockerBloc` directly. These must be verified and updated if necessary.
