# Idea: Lock Application on Device Screen Lock (AW-2349)

## Summary

MFA spec Section 8 requires: "if the device is locked (lock screen) → lock immediately." Currently the app only locks via timer expiration (`TimerService`) or background timeout (`shouldLockOnResume` check on `AppLifecycleState.resumed`). There is no detection of the device screen being locked.

The goal is to:
1. **Detect** device screen lock events on all platforms (Android, iOS, macOS, Windows) via native platform APIs.
2. **Lock** the MFA locker immediately when the screen lock is detected, by adding an EventChannel to the `biometric_cipher` plugin and wiring it through a new `ScreenLockService` into `LockerBloc`.

---

## Affected Repository

`github.com/AdguardTeam/mfa_locker` — contains two sub-packages:
- `packages/biometric_cipher/` — Flutter plugin wrapping platform biometric + crypto APIs
- `lib/` — Dart locker logic (`MFALocker`, security providers, exception types)
- `example/` — Demo app with BLoC architecture (UI → BLoC → Repository → MFALocker)

---

## Platform Detection Mechanisms

| Platform | Signal | API | Prompt shown? |
|----------|--------|-----|---------------|
| Android | Screen off | `BroadcastReceiver` for `ACTION_SCREEN_OFF` | No |
| iOS | Data protection unavailable | `UIApplication.protectedDataWillBecomeUnavailableNotification` | No |
| macOS | Screen locked | `DistributedNotificationCenter` for `com.apple.screenIsLocked` | No |
| Windows | Session locked | `WTSRegisterSessionNotification` + `WM_WTSSESSION_CHANGE` / `WTS_SESSION_LOCK` | No |

**Why `ACTION_SCREEN_OFF` on Android (not `KeyguardManager.isKeyguardLocked`)?**
`ACTION_SCREEN_OFF` fires synchronously when the screen turns off — no polling, no latency. For a security app, locking on every screen-off is the conservative and correct approach. Even if the device has no lock screen configured, the MFA app should still protect its data. `KeyguardManager.isKeyguardLocked()` is a poll-based API and would introduce a race condition (keyguard may not be locked yet when `ACTION_SCREEN_OFF` fires).

**Why `protectedDataWillBecomeUnavailable` on iOS (not Darwin notification)?**
`protectedDataWillBecomeUnavailableNotification` is a public, App Store-safe API that fires when data protection activates (device locking). The alternative (`com.apple.springboard.lockstate` Darwin notification) is a private API and risks App Store rejection.

**Edge case — Flutter engine suspended:** If the app is already suspended when the device locks, the EventChannel event may not be delivered. This is acceptable because the existing `shouldLockOnResume` mechanism in `TimerService` handles this case: when the app resumes, the timer will have expired and the app locks via `_onAppResumed`. The screen lock detection adds an *eagerness* path for when the app is in the foreground.

## B. `biometric_cipher` Native — Android

### Context

The Android plugin (`BiometricCipherPlugin.kt`) implements `FlutterPlugin` and `ActivityAware`. It already stores `flutterPluginBinding.applicationContext` during `onAttachedToEngine`. The `BroadcastReceiver` for `ACTION_SCREEN_OFF` must be registered with the application context (not the activity context) because the activity may not exist when the screen is off.

`ACTION_SCREEN_OFF` is a protected broadcast that can only be received by dynamically registered receivers — manifest registration is not supported. This aligns with the EventChannel lifecycle: register in `onListen`, unregister in `onCancel`.

**File**: new — `packages/biometric_cipher/android/src/main/kotlin/com/adguard/cryptowallet/biometric_cipher/handlers/ScreenLockStreamHandler.kt`

### Change 1: create `ScreenLockStreamHandler`

```kotlin
package com.adguard.cryptowallet.biometric_cipher.handlers

import android.content.BroadcastReceiver
import android.content.Context
import android.content.Intent
import android.content.IntentFilter
import io.flutter.plugin.common.EventChannel

class ScreenLockStreamHandler(
    private val applicationContext: Context,
) : EventChannel.StreamHandler {

    private var receiver: BroadcastReceiver? = null

    override fun onListen(arguments: Any?, events: EventChannel.EventSink?) {
        val screenOffReceiver = object : BroadcastReceiver() {
            override fun onReceive(context: Context?, intent: Intent?) {
                if (intent?.action == Intent.ACTION_SCREEN_OFF) {
                    events?.success(true)
                }
            }
        }

        val filter = IntentFilter(Intent.ACTION_SCREEN_OFF)
        applicationContext.registerReceiver(screenOffReceiver, filter)
        receiver = screenOffReceiver
    }

    override fun onCancel(arguments: Any?) {
        receiver?.let { applicationContext.unregisterReceiver(it) }
        receiver = null
    }
}
```

The receiver is created in `onListen` and destroyed in `onCancel`, matching the EventChannel lifecycle. If the Dart stream is cancelled (e.g., `ScreenLockService.stopListening()`), the native receiver is unregistered.

## C. `biometric_cipher` Native — iOS / macOS

### Context

iOS and macOS share code in `packages/biometric_cipher/darwin/Classes/` with `#if os(iOS)` / `#elseif os(macOS)` guards where needed. The screen lock signals differ:
- **iOS**: `UIApplication.protectedDataWillBecomeUnavailableNotification` — fires when data protection activates (device locks with passcode enabled)
- **macOS**: `com.apple.screenIsLocked` — distributed notification fired when the screen locks (Cmd+Ctrl+Q, auto-lock, sleep lock)

**File**: new — `packages/biometric_cipher/darwin/Classes/ScreenLockStreamHandler.swift`

### Change 1: create `ScreenLockStreamHandler`

```swift
#if os(iOS)
import UIKit
#elseif os(macOS)
import AppKit
#endif
import Flutter

class ScreenLockStreamHandler: NSObject, FlutterStreamHandler {
    private var eventSink: FlutterEventSink?

    func onListen(withArguments arguments: Any?, eventSink events: @escaping FlutterEventSink) -> FlutterError? {
        eventSink = events

        #if os(iOS)
        NotificationCenter.default.addObserver(
            self,
            selector: #selector(onScreenLocked),
            name: UIApplication.protectedDataWillBecomeUnavailableNotification,
            object: nil
        )
        #elseif os(macOS)
        DistributedNotificationCenter.default().addObserver(
            self,
            selector: #selector(onScreenLocked),
            name: NSNotification.Name("com.apple.screenIsLocked"),
            object: nil
        )
        #endif

        return nil
    }

    func onCancel(withArguments arguments: Any?) -> FlutterError? {
        #if os(iOS)
        NotificationCenter.default.removeObserver(self)
        #elseif os(macOS)
        DistributedNotificationCenter.default().removeObserver(self)
        #endif

        eventSink = nil
        return nil
    }

    @objc private func onScreenLocked() {
        eventSink?(true)
    }
}
```

`NotificationCenter` is used for iOS (app-local notifications), while `DistributedNotificationCenter` is used for macOS (system-wide notifications). The observer is added in `onListen` and removed in `onCancel`.

## D. `biometric_cipher` Native — Windows

### Context

The Windows plugin uses C++/WinRT. To detect session lock, it must:
1. Call `WTSRegisterSessionNotification(hwnd, NOTIFY_FOR_THIS_SESSION)` to subscribe to session change events
2. Handle `WM_WTSSESSION_CHANGE` window messages with `wParam == WTS_SESSION_LOCK`

The plugin has access to the registrar, which provides the window handle via `registrar->GetView()->GetNativeWindow()`. The `RegisterTopLevelWindowProcDelegate` method allows hooking into the window procedure to intercept messages.

**File**: new — `packages/biometric_cipher/windows/include/biometric_cipher/handlers/screen_lock_stream_handler.h`

### Change 1: create `ScreenLockStreamHandler` class declaration

```cpp
#pragma once

#include <flutter/event_channel.h>
#include <flutter/event_stream_handler_functions.h>
#include <flutter/plugin_registrar_windows.h>
#include <windows.h>
#include <wtsapi32.h>

class ScreenLockStreamHandler {
public:
    explicit ScreenLockStreamHandler(flutter::PluginRegistrarWindows* registrar);
    ~ScreenLockStreamHandler();

    std::unique_ptr<flutter::StreamHandler<flutter::EncodableValue>> CreateStreamHandler();

private:
    flutter::PluginRegistrarWindows* registrar_;
    std::unique_ptr<flutter::EventSink<flutter::EncodableValue>> event_sink_;
    int window_proc_delegate_id_ = -1;

    std::optional<LRESULT> HandleWindowMessage(HWND hwnd, UINT message, WPARAM wparam, LPARAM lparam);
    void RegisterWindowProc();
    void UnregisterWindowProc();
};
```

**File**: new — `packages/biometric_cipher/windows/handlers/screen_lock_stream_handler.cpp`

### Change 2: implement `ScreenLockStreamHandler`

```cpp
#include "biometric_cipher/handlers/screen_lock_stream_handler.h"

#pragma comment(lib, "Wtsapi32.lib")

ScreenLockStreamHandler::ScreenLockStreamHandler(
    flutter::PluginRegistrarWindows* registrar)
    : registrar_(registrar) {}

ScreenLockStreamHandler::~ScreenLockStreamHandler() {
    UnregisterWindowProc();
}

std::unique_ptr<flutter::StreamHandler<flutter::EncodableValue>>
ScreenLockStreamHandler::CreateStreamHandler() {
    return std::make_unique<flutter::StreamHandlerFunctions<flutter::EncodableValue>>(
        // onListen
        [this](const flutter::EncodableValue* arguments,
               std::unique_ptr<flutter::EventSink<flutter::EncodableValue>>&& events)
            -> std::unique_ptr<flutter::StreamHandlerError<flutter::EncodableValue>> {
            event_sink_ = std::move(events);
            RegisterWindowProc();
            return nullptr;
        },
        // onCancel
        [this](const flutter::EncodableValue* arguments)
            -> std::unique_ptr<flutter::StreamHandlerError<flutter::EncodableValue>> {
            UnregisterWindowProc();
            event_sink_ = nullptr;
            return nullptr;
        });
}

void ScreenLockStreamHandler::RegisterWindowProc() {
    HWND hwnd = registrar_->GetView()->GetNativeWindow();
    WTSRegisterSessionNotification(hwnd, NOTIFY_FOR_THIS_SESSION);

    window_proc_delegate_id_ = registrar_->RegisterTopLevelWindowProcDelegate(
        [this](HWND hwnd, UINT message, WPARAM wparam, LPARAM lparam) {
            return HandleWindowMessage(hwnd, message, wparam, lparam);
        });
}

void ScreenLockStreamHandler::UnregisterWindowProc() {
    if (window_proc_delegate_id_ >= 0) {
        HWND hwnd = registrar_->GetView()->GetNativeWindow();
        WTSUnRegisterSessionNotification(hwnd);
        registrar_->UnregisterTopLevelWindowProcDelegate(window_proc_delegate_id_);
        window_proc_delegate_id_ = -1;
    }
}

std::optional<LRESULT> ScreenLockStreamHandler::HandleWindowMessage(
    HWND hwnd, UINT message, WPARAM wparam, LPARAM lparam) {
    if (message == WM_WTSSESSION_CHANGE && wparam == WTS_SESSION_LOCK) {
        if (event_sink_) {
            event_sink_->Success(flutter::EncodableValue(true));
        }
    }
    return std::nullopt;  // Let other handlers process the message
}
```

## E. Example App — ScreenLockService

### Context

The existing `TimerService` pattern provides a clean abstraction for lock triggers. `ScreenLockService` follows the same pattern: an abstract interface with a callback, injected into `LockerBloc`. The service wraps the `BiometricCipher.screenLockStream` and invokes the callback when a lock event is received.

**File**: new — `example/lib/core/services/screen_lock_service.dart`

### Change 1: create `ScreenLockService` interface and implementation

```dart
import 'dart:async';

import 'package:biometric_cipher/biometric_cipher.dart';

abstract class ScreenLockService {
  /// Set the callback to be invoked when the device screen is locked.
  set onScreenLockedCallback(void Function() onLock);

  /// Start listening for screen lock events.
  void startListening();

  /// Stop listening for screen lock events.
  void stopListening();

  /// Dispose of resources.
  void dispose();
}

class ScreenLockServiceImpl implements ScreenLockService {
  ScreenLockServiceImpl({required BiometricCipher biometricCipher})
      : _biometricCipher = biometricCipher;

  final BiometricCipher _biometricCipher;
  StreamSubscription<bool>? _subscription;
  void Function()? _onScreenLocked;

  @override
  set onScreenLockedCallback(void Function() onLock) => _onScreenLocked = onLock;

  @override
  void startListening() {
    _subscription?.cancel();
    _subscription = _biometricCipher.screenLockStream.listen((_) {
      _onScreenLocked?.call();
    });
  }

  @override
  void stopListening() {
    _subscription?.cancel();
    _subscription = null;
  }

  @override
  void dispose() {
    stopListening();
    _onScreenLocked = null;
  }
}
```

The service creates a new subscription in `startListening()` and cancels it in `stopListening()`. The callback pattern matches `TimerService.onLockCallback`.

## G. Example App — LockerBloc Integration

### Context

The `LockerBloc` already handles auto-lock via `_timerService` (timer-based) and `_onAppResumed` (background timeout). Screen lock detection follows the same pattern: a service notifies the BLoC, which then calls `_lockerRepository.lock()`.

**Key design decision**: The `_onScreenLocked` handler **bypasses** the `BiometricOperationState` guard that `_onLockRequested` enforces. A physical device lock is an explicit security action — the OS has already dismissed any biometric dialog. The app must lock immediately regardless of in-progress biometric flows.

**File**: `example/lib/features/locker/bloc/locker_event.dart`

### Change 3: add callback and handler

```dart
/// Called by ScreenLockService when the device screen is locked.
void _onScreenLockDetected() {
  if (!isClosed && state.status == LockerStatus.unlocked) {
    add(const LockerEvent.screenLocked());
  }
}

/// Handles the screen lock event.
///
/// Bypasses the [BiometricOperationState] guard — a physical device lock
/// is an explicit security action that overrides any in-progress biometric flow.
/// The OS has already dismissed the biometric dialog.
Future<void> _onScreenLocked(
  _ScreenLocked event,
  Emitter<LockerState> emit,
) async {
  if (state.status != LockerStatus.unlocked) return;
  await _lockerRepository.lock();
}
```

### Change 4: start/stop listening on state transitions

In the existing handler that starts `_timerService` on unlock (where `_timerService.startTimer()` is called), add:

```dart

```

This ensures the native listener is only active when the locker is unlocked, avoiding unnecessary native event processing.

## Acceptance Criteria

1. On Android, the locker transitions to `locked` state when `ACTION_SCREEN_OFF` fires while the locker is `unlocked`.
2. On iOS, the locker transitions to `locked` state when `protectedDataWillBecomeUnavailableNotification` fires while the locker is `unlocked`.
3. On macOS, the locker transitions to `locked` state when the `com.apple.screenIsLocked` distributed notification fires while the locker is `unlocked`.
4. On Windows, the locker transitions to `locked` state when `WTS_SESSION_LOCK` is received while the locker is `unlocked`.
5. Screen lock detection does **not** trigger a biometric prompt or any user-visible system dialog.
6. Screen lock immediately locks the app even if a biometric operation is in progress (bypasses `BiometricOperationState` guard).
7. Screen lock events are ignored when the locker is already `locked`, `notInitialized`, or in any non-`unlocked` state.
8. The native listener is only active when the locker is in `unlocked` state (started on unlock, stopped on lock).
9. Existing timer-based auto-lock and `shouldLockOnResume` behavior continues to work unchanged.
10. If the Flutter engine is suspended when the device locks, the existing `shouldLockOnResume` mechanism catches it on app resume.
11. macOS full-screen transitions do not trigger false screen lock events (the distributed notification is independent of app lifecycle).
