# AW-2349-phase-7: Screen Lock Detection — DI Wiring + `screenLocked` Event

Status: PRD_READY

## Context / Idea

MFA spec Section 8 requires: "if the device is locked (lock screen) → lock immediately." The full feature (AW-2349) adds native screen lock detection across Android, iOS, macOS, and Windows via an `EventChannel` in the `biometric_cipher` plugin, then wires it through the example app's architecture.

Phases 1–6 are complete:
- **Phase 1:** Dart plugin layer — `screenLockStream` on `BiometricCipherPlatform`, `MethodChannelBiometricCipher`, and `BiometricCipher`.
- **Phase 2:** Android native handler (`ScreenLockStreamHandler.kt` + `BroadcastReceiver` for `ACTION_SCREEN_OFF`).
- **Phase 3:** iOS/macOS native handler (`ScreenLockStreamHandler.swift` using `protectedDataWillBecomeUnavailableNotification` / `com.apple.screenIsLocked`).
- **Phase 4:** Windows native handler (`ScreenLockStreamHandler.cpp/.h` using `WTS_SESSION_LOCK`).
- **Phase 5:** Plugin-level unit tests for `screenLockStream`.
- **Phase 6:** `ScreenLockService` abstract interface and `ScreenLockServiceImpl` in `example/lib/core/services/screen_lock_service.dart`.

**Phase 7 (this phase)** wires `ScreenLockService` through the existing factory-based DI graph and declares the Freezed event that `LockerBloc` will handle in Phase 8. The example app DI pattern is: `RepositoryFactory` creates services → `BlocFactory` injects them into BLoCs → `main.dart` connects the factories. `ScreenLockService` follows the same path that `TimerService` already uses today.

Key design points from `docs/idea-2349.md` Section F:
- `RepositoryFactoryImpl` must create a `BiometricCipher` instance to pass to `ScreenLockServiceImpl`. Unlike what the idea doc assumes, inspection of `repository_factory.dart` shows no `_biometricCipher` field currently exists in `RepositoryFactoryImpl` — it must be introduced here.
- `RepositoryFactory.dispose()` must call `_screenLockService.dispose()` to release the stream subscription.
- `BlocFactoryImpl` receives `ScreenLockService` as a constructor parameter and stores it as a private field (mirrors `TimerService`). Passing `screenLockService` into the `LockerBloc(...)` call is **deferred to Phase 8** because `LockerBloc` does not yet accept that parameter.
- `LockerEvent.screenLocked()` is a zero-argument Freezed factory — same style as existing events.
- Code generation (`make g`) is required after adding the new Freezed factory.

### Affected files (Phase 7 only)

| File | Change |
|------|--------|
| `example/lib/di/factories/repository_factory.dart` | Add `BiometricCipher` field, create `ScreenLockServiceImpl` in `init()`, expose via getter, dispose in `dispose()` |
| `example/lib/di/factories/bloc_factory.dart` | Add `ScreenLockService` constructor parameter and field; do NOT pass to `LockerBloc(...)` yet (deferred to Phase 8) |
| `example/lib/main.dart` | Pass `repositoryFactory.screenLockService` to `BlocFactoryImpl` |
| `example/lib/features/locker/bloc/locker_event.dart` | Add `const factory LockerEvent.screenLocked() = _ScreenLocked;` |
| `example/lib/features/locker/bloc/locker_event.freezed.dart` (generated) | Regenerated by `make g` |

Phase 8 adds the actual `LockerBloc` handler, passes `screenLockService` into `LockerBloc(...)`, and handles start/stop listening and dispose.

## Goals

1. Introduce a `BiometricCipher` instance in `RepositoryFactoryImpl` and use it to create `ScreenLockServiceImpl` during `init()`.
2. Expose `ScreenLockService` via a getter on both the `RepositoryFactory` abstract interface and `RepositoryFactoryImpl`.
3. Ensure `RepositoryFactoryImpl.dispose()` calls `_screenLockService.dispose()`.
4. Add `ScreenLockService` as a required constructor parameter and private field to `BlocFactoryImpl` (after `TimerService`). The field is stored but not yet forwarded to `LockerBloc(...)`.
5. Wire `repositoryFactory.screenLockService` into `BlocFactoryImpl` construction in `main.dart`.
6. Declare `LockerEvent.screenLocked()` as a zero-argument Freezed factory in `locker_event.dart`.
7. Run `cd example && make g` to regenerate Freezed code without errors.
8. Pass `cd example && fvm flutter analyze --fatal-warnings --fatal-infos --no-pub .` with zero warnings or infos.

---

## User Stories

**As the example app DI layer**, I want `ScreenLockService` to be instantiated once in `RepositoryFactoryImpl` and reachable via a getter, so that BLoC construction in `BlocFactoryImpl` can receive it as a proper dependency without coupling to the plugin directly.

**As a future BLoC integration author (Phase 8)**, I want `BlocFactoryImpl` to already hold `_screenLockService` as a field, so that I can wire it into the `LockerBloc(...)` call in Phase 8 without touching DI plumbing again.

**As a future test author**, I want `LockerEvent.screenLocked()` to be available in the Freezed union, so that I can assert correct BLoC behavior when the event is dispatched.

**As a developer reviewing the codebase**, I want `ScreenLockService` to follow the same DI path as `TimerService` (factory getter → BlocFactory param → BLoC constructor), so that the feature requires no new architectural concepts to understand.

## Main Scenarios

### Scenario 1 — Normal app startup wiring
- `RepositoryFactoryImpl.init()` is called.
- `_biometricCipher` is instantiated (`BiometricCipher()`).
- `_screenLockService` is instantiated with `biometricCipher: _biometricCipher`.
- `BlocFactoryImpl` is created with `screenLockService: repositoryFactory.screenLockService`.
- `BlocFactoryImpl` stores the service as `_screenLockService`. The `LockerBloc(...)` call inside `BlocFactoryImpl.lockerBloc` does not receive it yet (Phase 8 handles that).

### Scenario 2 — App teardown
- `repositoryFactory.dispose()` is called.
- `_screenLockService.dispose()` is called, cancelling the native event subscription.
- `_lockerRepository.dispose()` is called as before.
- No resource leaks.

### Scenario 3 — `LockerEvent.screenLocked()` used in Phase 8
- The event is declared in `locker_event.dart` and code-generated in Phase 7.
- `LockerBloc` registers an `on<_ScreenLocked>` handler in Phase 8.
- Phase 7 contributes only the event declaration; behavior is Phase 8's scope.

### Scenario 4 — Missing `_biometricCipher` in `RepositoryFactoryImpl`
- `RepositoryFactoryImpl` currently has no `BiometricCipher` field.
- A new `late final BiometricCipher _biometricCipher` field is added.
- It is instantiated as `_biometricCipher = BiometricCipher()` inside `init()`.
- `ScreenLockServiceImpl(biometricCipher: _biometricCipher)` follows in the same `init()` block.

### Scenario 5 — Code generation succeeds
- `LockerEvent.screenLocked()` is added to the sealed Freezed class.
- `cd example && make g` regenerates `locker_event.freezed.dart` (and any `.g.dart` if applicable) without errors.
- `fvm flutter analyze` reports zero issues.

## Constraints and Assumptions

- **Scope is 4 files + code generation:** Phase 7 touches `repository_factory.dart`, `bloc_factory.dart`, `main.dart`, `locker_event.dart` only (plus the auto-generated output of `make g`). No other files change.
- **`BiometricCipher` instance must be created in `RepositoryFactoryImpl.init()`:** `RepositoryFactoryImpl` currently has no `_biometricCipher` field. The instance must be created in `init()` before `ScreenLockServiceImpl`, because `init()` is already the initialization entry point for `_timerService` and biometric configuration.
- **Deferred BLoC pass-through (resolved):** Phase 7 adds `ScreenLockService` as a constructor parameter and private field to `BlocFactoryImpl`, but does NOT pass it to the `LockerBloc(...)` call inside `BlocFactoryImpl.lockerBloc`. That wiring is strictly Phase 8 scope. This avoids a compile error because `LockerBloc` does not yet accept the parameter.
- **No new tests in this phase:** Unit tests covering DI wiring are integration-level; they are not scoped to Phase 7.
- **Code generation required:** `locker_event.dart` uses Freezed (`@freezed sealed class`). Adding a factory requires running `make g` to regenerate the `.freezed.dart` file.
- **`const` on `BlocFactoryImpl` constructor:** The current `BlocFactoryImpl` uses `const BlocFactoryImpl(...)`. Adding `ScreenLockService` (an abstract interface, not a `const`-compatible type) removes the ability to use a `const` constructor. The `const` keyword must be removed from the constructor.
- **Constructor parameter order:** `ScreenLockService` is added after `TimerService` in both the constructor parameter list and the field declaration list of `BlocFactoryImpl`, following existing ordering conventions.
- **`RepositoryFactory.dispose()` is `async`:** The existing `dispose()` is `Future<void>`. `_screenLockService.dispose()` is synchronous — it should be called before `await _lockerRepository?.dispose()`.
- **Follows `TimerService` pattern exactly:** Field name `_screenLockService`, getter name `screenLockService`, parameter name `screenLockService`.
