# AW-2349 Phase 7 Research: DI Wiring + `screenLocked` Event

## Resolved Questions

**Q1 — BLoC wiring deferral**
`BlocFactoryImpl` stores `_screenLockService` as a field in Phase 7 but does NOT pass it to `LockerBloc(...)`. That wiring is deferred to Phase 8 (when `LockerBloc` gains its `screenLockService` parameter). This is confirmed by the PRD.

**Q2 — `const` check on `BlocFactoryImpl`**
Before removing `const`, all usages of `BlocFactoryImpl(` were searched across the entire repo. Results:
- `example/lib/di/factories/bloc_factory.dart:18` — the constructor declaration itself (`const BlocFactoryImpl({`)
- `example/lib/main.dart:39` — the single instantiation site (`final blocFactory = BlocFactoryImpl(`)
- Docs/phase/idea files — text references only, not Dart code

There are no `const BlocFactoryImpl(...)` call-sites in any Dart source files. The `const` keyword appears only on the constructor declaration itself, not at any construction site. Removing `const` from the constructor is safe with zero breakage risk.

**Q3 — `_biometricCipher` existence in `RepositoryFactoryImpl`**
`_biometricCipher` does NOT exist in `RepositoryFactoryImpl`. The file contains:
- `LockerRepository? _lockerRepository` — nullable, lazy via `??=`
- `TimerService? _timerService` — nullable, set in `init()`

The only `BiometricCipher`-related code is a call to `lockerRepository.configureBiometricCipher(...)` inside `init()`. There is no `BiometricCipher` field or import. A new `late final BiometricCipher _biometricCipher` field must be introduced.

## Phase-Specific Limitations and Risks

1. **`_biometricCipher` initialization order in `init()`**: The field must be set before `_screenLockService` is constructed. Both must come before (or can be independent of) `configureBiometricCipher(...)`. Suggested order:
   - `_timerService = ...` (existing)
   - `_biometricCipher = BiometricCipher()` (new)
   - `_screenLockService = ScreenLockServiceImpl(biometricCipher: _biometricCipher)` (new)
   - `await lockerRepository.configureBiometricCipher(...)` (existing)

2. **`late final` vs nullable for `_biometricCipher` and `_screenLockService`**: Unlike `_timerService` and `_lockerRepository` which are nullable (`?`), the PRD specifies `late final` for the new fields. This matches the intent that they are always set in `init()`. If `init()` is not called and the getter is accessed, a `LateInitializationError` will be thrown (similar to the explicit `StateError` guard on `timerService`). The PRD says to use `late final`. For consistency with `timerService`, adding an explicit `StateError` guard on the getter (via a helper or tracking field) could be done, but the PRD example shows `late final` without a guard — follow the PRD.

3. **`ScreenLockService.dispose()` is synchronous**: `dispose()` on the service is `void` (not `Future<void>`). In `RepositoryFactoryImpl.dispose()` (which is `Future<void>`), call it as a plain synchronous call before `await _lockerRepository?.dispose()`.

4. **No `const BlocFactoryImpl(...)` at call sites**: Confirmed — only `main.dart` constructs it without `const`. Removing `const` from the constructor declaration has zero impact on callers.

5. **`LockerBloc` does not accept `screenLockService` yet**: Phase 7 must NOT pass `_screenLockService` to `LockerBloc(...)`. The `lockerBloc` getter in `BlocFactoryImpl` remains unchanged except for the removal of `const` on the constructor. Attempting to pass it now would cause a compile error.

6. **`make g` scope**: Freezed code-gen runs on files annotated with `@freezed`. Adding `LockerEvent.screenLocked()` to `locker_event.dart` (which is `part of 'locker_bloc.dart'`) will regenerate `locker_bloc.freezed.dart`. The `bloc_factory.dart` file is NOT a Freezed file so it has no gen impact.

7. **`_timerService` nullable vs `late final` inconsistency**: Existing `_timerService` is `TimerService?` (nullable) with a guard. The PRD specifies `late final` for new fields. This is an intentional difference in style — follow the PRD for `_biometricCipher` and `_screenLockService`.
