# Idea: Support list of initial entries in `init` and optional `EntryId` in `addEntry` (AW-2479)

## Summary

The `EncryptedStorage.init` method currently accepts exactly one initial entry (`EntryMeta` + `EntryValue`) with an auto-generated UUID. The consumer app needs to pass **multiple** initial entries and optionally specify their IDs. Similarly, `addEntry` always auto-generates a UUID — it needs an optional `EntryId` parameter so callers can supply a specific ID when needed.

---

## Approach

### New type: `NewEntry`

New file `lib/storage/models/domain/new_entry.dart`:

```dart
class NewEntry implements Erasable {
  final EntryId? id;
  final EntryMeta meta;
  final EntryValue value;

  const NewEntry({this.id, required this.meta, required this.value});

  @override
  bool get isErased => meta.isErased && value.isErased;

  @override
  void erase() {
    meta.erase();
    value.erase();
  }
}
```

Bundles `(EntryId?, EntryMeta, EntryValue)` for the `init` list. Implements `Erasable` so it integrates with `_executeWithCleanup`.

### New exception: `duplicateEntry`

Add `duplicateEntry` to `StorageExceptionType` enum and a factory to `StorageException`, so `addEntry` with an explicit ID that already exists throws a typed error.

## Implementation details

### `EncryptedStorageImpl.init`

- Loop over `initialEntries`, use `entry.id?.value ?? _generateEntryId()` for each entry.
- Validate no duplicate explicit IDs in the list before encrypting. Throw `StorageException` if duplicates found.
- Empty list is allowed — creates storage with master key, salt, HMAC, and empty entries array.

### `EncryptedStorageImpl.addEntry`

- Accept optional `EntryId? id` parameter.
- Use `id ?? EntryId(_generateEntryId())` instead of always generating.
- When `id` is explicitly provided, check for duplicates in existing entries — throw `StorageException.duplicateEntry` if found.
- Duplicate check only runs when `id != null` (UUID v4 collisions are astronomically improbable).

### `MFALocker.init`

- `erasables: [passwordCipherFunc, ...initialEntries]` — spread flattens `NewEntry` objects into the erasable list.
- No changes to `_metaCache` logic — `loadAllMetaIfLocked` after init reads meta from storage fresh.

### `MFALocker.write`

- Pass optional `id` through to `_storage.addEntry(id: id, ...)`.
- No changes to `_metaCache` logic.

### Example app

- `LockerRepositoryImpl.init` wraps single entry in `[NewEntry(meta: entryMeta, value: entryValue)]`.
- `addEntry`/`addEntryWithBiometric` unchanged (don't use explicit IDs).

## Key design decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| New type for init list | `NewEntry` class | Bundles 3 fields, implements `Erasable`, one-type-per-file |
| `NewEntry` for `addEntry`? | No — just add optional `EntryId? id` | Single-entry method; flat params are simpler |
| Empty init list | Allowed | More flexible; storage with only master key + HMAC is valid |
| Duplicate explicit ID | Throw `StorageException.duplicateEntry` | `addEntry` = "add new"; upsert belongs to `updateEntry` |
| Duplicate check scope | Only when `id != null` | UUID v4 collisions are astronomically improbable |
